\section{Module \texttt{dialog}}

Le module \texttt{dialog} fournit une API de boîtes d'alerte synchrones ou asynchrones basée sur \texttt{GtkAlertDialog}.

\subsection{Macros de Configuration}

\begin{docbox}{Macros \texttt{ALERT\_BUTTON\_CONFIG\_DEFAULT} et \texttt{ALERT\_DIALOG\_CONFIG\_DEFAULT}}
\begin{lstlisting}[language=C]
#define ALERT_BUTTON_CONFIG_DEFAULT ((alert_button_config){ \
    .label = NULL, \
    .appearance = NULL, \
})

#define ALERT_DIALOG_CONFIG_DEFAULT ((alert_dialog_config){ \
    .parent = NULL, \
    .message = NULL, \
    .detail = NULL, \
    .buttons = NULL, \
    .button_count = 0, \
    .default_button = -1, \
    .cancel_button = -1, \
    .modal = true, \
    .on_response = NULL, \
    .user_data = NULL, \
})
\end{lstlisting}
\end{docbox}

Les macros définissent une alerte sûre par défaut, avec des index invalides explicites pour les boutons. Les pointeurs \texttt{NULL} empêchent les accès à des chaînes ou callbacks non initialisés. Cette approche défensive réduit les risques de segmentation fault dans les chemins d'interaction utilisateur.

L'usage de valeurs sentinelles (\texttt{-1}) pour \texttt{default\_button} et \texttt{cancel\_button} évite les index hors limites. C'est une pratique robuste pour prévenir les erreurs de sélection côté UI.

\subsection{Fonctions API}

\begin{lstlisting}[language=C]
void show_alert_dialog(const alert_dialog_config *config) {
    GtkAlertDialog *dialog;

    if (config == NULL || config->message == NULL) {
        return;
    }

    dialog = gtk_alert_dialog_new("%s", config->message);
    if (config->detail != NULL) {
        gtk_alert_dialog_set_detail(dialog, config->detail);
    }

    if (config->on_response != NULL) {
        gtk_alert_dialog_choose(dialog, config->parent, NULL, on_alert_response, ctx);
    } else {
        gtk_alert_dialog_show(dialog, config->parent);
    }

    g_object_unref(dialog);
}
\end{lstlisting}

La fonction choisit automatiquement un mode asynchrone si un callback est fourni, sinon un mode simple d'affichage.

Ce design permet de couvrir deux besoins: une alerte informative rapide et une alerte interactive qui remonte un résultat utilisateur. L'API reste compacte tout en masquant la complexité asynchrone de GTK.
